# OpenSuper 讲法提纲（中文）

这份文档回答的不是“OpenSuper 怎么用”，而是：

> **当我们要介绍 `pzy560117/opensuper` 时，到底应该把它讲成什么。**

如果你需要更接近原始分享口播的版本，读 [presentation-transcript.zh.md](presentation-transcript.zh.md)。本文件不是逐段口播稿，而是已经收束过主轴的讲法提纲。

## 1. 先定调：这不是“工具介绍”

最容易把 OpenSuper 讲浅的方式，是把它讲成：

- 一个把 OpenSpec 和 Superpowers 串起来的工具；
- 一个 CLI；
- 一组 skills；
- 一套 Prompt。

这些都不算错，但都不够。

更准确的定义应该是：

> OpenSuper 是作者围绕真实 AI Coding 痛点做出来的一套**工程系统**，同时也是一套把方法论、规则、状态和知识沉淀为仓库事实的**文档系统**。

所以介绍它时，必须同时讲两条线：

1. **工程线**：它怎样把 AI Coding 变成可恢复、可验证、可归档的执行系统；
2. **文档线**：它怎样把经验、规则、讲法和知识固化成可复用的文档表面。

## 2. 整个分享的总主线

如果只保留一条主线，我建议这样讲：

1. AI Coding 最大的问题，不是第一次写不出代码，而是后续流程失控。
2. OpenSpec、Superpowers、规则文件、测试、Review 都有价值，但单独使用是散的。
3. OpenSuper 做的事，是把它们接成一条有 shared state 的工程链。
4. 这条链不仅体现在代码和脚本上，也体现在仓库文档体系上。
5. 所以 OpenSuper 不是一个“命令集合”，而是一套工程系统加文档系统。

## 3. 第一条线：工程系统应该怎么讲

### 3.1 起点：它不是从“我要做个工具”开始的

要先把问题讲出来：

- OpenSpec 擅长 WHAT；
- Superpowers 擅长 HOW；
- 规则文件能约束行为；
- 测试、Lint、Review 能校验结果；

但它们之间有缝：

- 会话断掉了怎么恢复；
- spec 和实现怎么对齐；
- 阶段怎么判断；
- 哪些选择必须由人来拍板；
- verify 和 archive 怎么不靠记忆收尾。

所以 OpenSuper 的起点不是“我想封装几个命令”，而是：

> 我需要一条稳定轨道，把 WHAT、HOW、状态、验证和归档接起来。

### 3.2 工程系统的四层结构

介绍它做了什么时，不要按文件讲，建议按四层讲。

#### 第一层：阶段化主链

OpenSuper 先定义了一条五阶段主流程：

- open
- design
- build
- verify
- archive

它的意义不是“阶段名字很好看”，而是：AI 不再靠临场猜下一步，而是被明确放进一个有前后关系的工程链。

#### 第二层：状态机

OpenSuper 真正拉开差距的地方，是它不只写流程说明，而是把流程状态落进 `.opensuper.yaml`，并用脚本维护：

- `opensuper-state.sh`
- `opensuper-guard.sh`
- `opensuper-archive.sh`
- `opensuper-yaml-validate.sh`

这层要重点讲，因为它回答的是：

- 为什么能断点恢复；
- 为什么能继续；
- 为什么不会每次都重新扫项目；
- 为什么阶段不是靠猜，而是靠状态。

#### 第三层：脚本级硬约束

OpenSuper 不是只给建议，而是把关键点升级成约束：

- `isolation`
- `build_mode`
- `verification_report`
- `branch_status`
- `build_command / verify_command`

这层最适合强调一句话：

> OpenSuper 把“容易忘”变成“忘不了”，把“看起来完成”变成“必须证明完成”。

#### 第四层：产品化与工程质量

OpenSuper 不停在“我自己本地能用”，而是继续长成了一个可发布产品：

- `init / status / doctor / update`
- JSON 输出
- 测试
- GitHub Actions
- release 流程
- changelog

这说明它已经不是个人工作流草稿，而是一个可安装、可验证、可维护的仓库产品。

### 3.3 为什么这条工程线尤其适合 C/C++ 和 TS

这里不要顺带一句带过，要明确说：

- **C/C++** 最需要的是边界、构建链、平台差异、验证闭环；
- **TS** 最需要的是收敛改动范围、控制上下文漂移、把类型检查和测试纳入 verify。

所以 OpenSuper 的工程线，最适合拿来解释：

- 为什么 AI Coding 不能只看“会不会生成”；
- 为什么更重要的是“有没有轨道和护栏”。

## 4. 第二条线：文档系统应该怎么讲

这是你现在最该补出来、也最容易被忽略的一条线。

OpenSuper 不只是一个工程系统，它还逐步长出了一套完整文档系统。这个文档系统不是附属物，而是工程本体的一部分。

### 4.1 文档系统里都有什么

当前仓库里，至少有这几层文档表面：

- [README-zh.md](../README-zh.md)：对外总览入口
- [AGENTS.md](../AGENTS.md)：短导航、目录地图、全局边界
- [CLAUDE.md](../CLAUDE.md)：项目级执行规约
- [guide.zh.md](guide.zh.md)：方法论映射
- [origin-and-evolution.zh.md](origin-and-evolution.zh.md)：项目起源与演进
- [presentation-transcript.zh.md](presentation-transcript.zh.md)：原始口播整理稿
- 本文件 [talk-track.zh.md](talk-track.zh.md)：对外讲法提纲
- `CHANGELOG.md`：演进事实

### 4.2 为什么文档系统不是可有可无

讲这个项目时，必须点破一个常见误区：

> 很多人以为工程系统是代码，文档只是补充；但在 OpenSuper 里，文档本身就是工程的一部分。

原因很简单：

- 没有 README，就没有对外入口；
- 没有 AGENTS / CLAUDE，就没有规则表面；
- 没有 guide，就没有方法论解释层；
- 没有 origin / talk-track / transcript，就没有“怎么讲”和“怎么沉淀”的层。

也就是说，OpenSuper 不只是把 AI Coding 工程化了，还把“AI Coding 工程该怎么解释、怎么传播、怎么复用”也工程化了。

### 4.3 文档系统的真正价值

这套文档系统至少解决三件事：

1. **让人能理解**：不是只给命令，还说明动机、结构、边界和适用场景；
2. **让 Agent 能遵守**：规则文件和技能文件形成可执行约束；
3. **让经验能回流**：讲法、起源、原稿、方法论都能持续沉淀，不留在聊天里。

所以你在分享时，可以直接说：

> OpenSuper 不只是把执行链做出来了，还把解释链和知识回流链也做出来了。

## 5. 最推荐的讲法结构

如果现在要对外讲“这套工程和文档内容”，我建议用下面这套结构。

### 开场

不要从工具名字开始，直接从问题开始：

> AI Coding 最麻烦的地方，不是第一次写代码，而是需求、设计、实现、验证、归档之间没有一条稳定轨道。OpenSuper 就是在解决这件事。

### 第一部分：工程系统

讲四层：

1. 五阶段主链
2. 状态机
3. 守护脚本与硬约束
4. CLI / tests / release

### 第二部分：文档系统

讲七类文档：

1. README
2. AGENTS
3. CLAUDE
4. Guide
5. Origin
6. Transcript
7. Talk-track

强调：这些文档不是附属品，而是这套工程的一部分。

### 第三部分：作者方法论

讲三点：

1. 不重写 OpenSpec / Superpowers，只做组合调度；
2. 不满足于“能跑”，而是追求“能恢复、能验证、能归档、能复用”；
3. 不把经验留在聊天里，而是继续沉到规则、脚本、测试和文档中。

### 第四部分：为什么对 C/C++ 和 TS 特别有价值

这里单独拉出来，不要在结尾顺带一句。

## 6. 一段适合正式分享的定义

下面这段可以直接拿去讲：

> OpenSuper 不是一个单纯把 OpenSpec 和 Superpowers 接起来的工具，而是我围绕真实 AI Coding 痛点做出来的一套工程系统。它一边把需求、设计、计划、实现、验证、归档接成有 shared state 的执行链；另一边又把规则、方法论、起源、讲法和经验沉淀成仓库文档系统。它真正想证明的不是 AI 能不能写代码，而是 AI 写代码这件事能不能被工程化，能不能被解释，能不能被复用。尤其在 C/C++ 和 TypeScript 这类高度依赖边界与验证闭环的项目里，这一点非常关键。

## 7. 这套工程不应该怎么讲

不要讲成下面几种样子：

- 不要讲成“工具对比”
- 不要讲成“Prompt 工程”
- 不要讲成“我封装了几个命令”
- 不要只讲 workflow，不讲状态机
- 不要只讲代码，不讲文档系统

这些讲法都会把 OpenSuper 的价值讲窄。

## 8. 最后一句应该落在哪里

最后一句不要落在“欢迎大家使用”，而应该落在：

> OpenSuper 证明了一件事：AI Coding 真正需要的，不只是强模型和强工具，而是一套把执行链、规则链和知识链接起来的工程系统。
